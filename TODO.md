# Protocol-correct outbound scheduling for rumqttc

This note captures follow-up work discovered while fixing graceful disconnect behavior for issue #1031.
It intentionally does not depend on `TODO.md`.

## Motivation

`Client::disconnect()` has been changed toward graceful shutdown semantics: once the event loop observes the
disconnect request, it enters a draining mode, completes previously accepted outbound protocol state, then sends
MQTT `DISCONNECT` as the final control packet.

The current event loop still uses broad request-channel gating when outbound QoS 1/2 publish inflight is full or a
packet-id collision is active. The follow-up work is to replace that with protocol-aware scheduling rather than adding
more narrow special cases.

## Protocol constraints

MQTT 5 Receive Maximum is a `PUBLISH`-only flow-control semantic:

- A client must not send more than Receive Maximum unacknowledged QoS 1/2 `PUBLISH` packets.
- A client must not delay packets other than `PUBLISH` because it has sent Receive Maximum `PUBLISH` packets
  without receiving acknowledgements.
- The same principle applies symmetrically to the server.
- QoS 0 `PUBLISH` is not counted by Receive Maximum.

For v4, there is no MQTT 5 Receive Maximum, but rumqttc's local inflight limit serves the same practical role for
outbound QoS 1/2 publish flow control. It should not be used as a broad reason to delay protocol progress packets.

`DISCONNECT` has terminal semantics:

- Graceful disconnect should finish the accepted work that the API promises to drain, then send and flush
  MQTT `DISCONNECT`.
- After actually sending MQTT `DISCONNECT`, the client must send no more MQTT control packets on that connection.
- An abort/drop transport path is distinct from MQTT `DISCONNECT` and may trigger Will publication.

MQTT acknowledgement and QoS 2 continuation packets are not Receive-Maximum-gated, but they do have ordering rules:

- `PUBACK` must be sent in the order of the corresponding inbound QoS 1 `PUBLISH` packets.
- `PUBREC` must be sent in the order of the corresponding inbound QoS 2 `PUBLISH` packets.
- `PUBREL` must be sent in the order of the corresponding successful `PUBREC` packets received for outbound QoS 2
  publishes.
- `PUBCOMP` acknowledges the received `PUBREL` packet identifier. MQTT 5 section 4.6 does not impose the same
  explicit ordering rule for `PUBCOMP`, but it is still protocol-progress traffic and not publish inflight traffic.

Packet identifiers must be preserved across acknowledgement flows:

- QoS 1 `PUBACK` uses the original QoS 1 `PUBLISH` packet identifier.
- QoS 2 `PUBREC` uses the original QoS 2 `PUBLISH` packet identifier.
- QoS 2 `PUBREL` uses the original `PUBLISH` / received `PUBREC` packet identifier.
- QoS 2 `PUBCOMP` uses the packet identifier from the received `PUBREL`.

## Why a narrow workaround is not enough

A one-slot FIFO lookahead can fix only this shape:

```text
publish inflight full
front of request channel = DISCONNECT
```

It does not solve the broader protocol scheduling problem:

```text
publish inflight full
front of request channel = blocked QoS 1/2 PUBLISH
next request = SUBSCRIBE / UNSUBSCRIBE / DISCONNECT / PUBACK / PUBREC / PUBREL / PUBCOMP
```

Strict global FIFO means the blocked publish delays later non-`PUBLISH` packets. That can be defended only as an
internal application queue limitation. It is not a correct model of MQTT Receive Maximum, which gates only further
QoS 1/2 `PUBLISH` packets.

Do not add more ad hoc "skip-the-line" special cases. They are difficult to reason about and risk hidden reordering
bugs. The right fix is a deliberate scheduler that separates request admission from wire scheduling.

## Design direction

Introduce an outbound scheduler with packet-class awareness.

The scheduler should distinguish at least:

- Publish-flow-controlled traffic: outbound QoS 1/2 `PUBLISH`, gated by MQTT 5 Receive Maximum or v4/v5 local
  inflight limits.
- Non-flow-controlled publishes: outbound QoS 0 `PUBLISH`, not Receive-Maximum-gated.
- Protocol-progress traffic: `PUBACK`, `PUBREC`, `PUBREL`, `PUBCOMP`, malformed-packet disconnects, and other packets
  generated by state-machine progress.
- Application control traffic: `SUBSCRIBE`, `UNSUBSCRIBE`, `PINGREQ`, `AUTH`, and `DISCONNECT`.

Expected scheduling behavior:

- A full QoS 1/2 publish inflight window blocks only additional outbound QoS 1/2 `PUBLISH` packets.
- Protocol-progress packets must be able to advance while publish inflight is full.
- Application control packets should be able to advance while publish inflight is full, subject to terminal
  disconnect and any explicit API ordering guarantees.
- `DISCONNECT` must remain terminal once sent.
- Graceful disconnect must drain the selected barrier set before sending MQTT `DISCONNECT`.
- Immediate disconnect must use a narrow shutdown/control path and must not wait for unresolved publish handshakes.
- A timeout-based graceful disconnect must start its timeout when the event loop observes the disconnect barrier and
  starts the drain phase, not when the client API merely accepts/enqueues the request.

## Ordering model

The refactor must explicitly define which ordering guarantees are preserved:

- Preserve FIFO for outbound application publishes by default.
- Do not let later QoS 0 publishes overtake earlier blocked QoS 1/2 publishes by default. MQTT 5 Receive Maximum does
  not gate QoS 0 `PUBLISH`, so bypass would be protocol-legal, but it changes API FIFO semantics and
  broker-observed publish order.
- Allow non-`PUBLISH` control/progress packets to pass QoS 1/2 publishes that are blocked only by publish inflight /
  Receive Maximum.
- Preserve required MQTT acknowledgement ordering for `PUBACK`, `PUBREC`, and `PUBREL`.
- Preserve tracked subscribe/unsubscribe completion semantics.
- Do not let non-`PUBLISH` control traffic reorder already-sent protocol state incorrectly.
- Document that rumqttc does not guarantee global wire FIFO across all API calls under publish flow-control pressure.
  It preserves protocol-correct ordering and class-specific ordering.

A single "non-PUBLISH queue" can be correct only if all producers insert into it in protocol-correct order. That is
fragile unless enqueueing is centralized in the state machine. A more robust approach is separate internal queues for
publish-flow-controlled work, protocol-progress work, and application control work, with one scheduler responsible for
selecting the next packet.

## Recommended policy

- Use class-based scheduling, not global FIFO wire scheduling.
- Preserve FIFO within outbound application publishes by default.
- Do not let QoS 0 publishes bypass earlier blocked QoS 1/2 publishes by default.
- Let non-`PUBLISH` protocol/control packets pass blocked QoS 1/2 publishes.
- Preserve MQTT-required ordering for `PUBACK`, `PUBREC`, and `PUBREL`.
- Preserve tracked `SUBSCRIBE` / `UNSUBSCRIBE` completion semantics.
- Make `disconnect_with_timeout()` a graceful drain barrier whose timeout starts when the event loop observes the
  barrier.
- Make `disconnect_now()` a separate shutdown/control signal, not a normal queued request.
- Share the scheduler structure between v4 and v5, with v5 additionally honoring broker Receive Maximum.

Short model:

```text
Publish lane: FIFO, flow-controlled.
Protocol-progress lane: always allowed when protocol-valid.
Control lane: allowed through publish flow control.
QoS 0: legal to bypass Receive Maximum, but do not bypass by default.
Graceful disconnect: barrier + drain + DISCONNECT.
Immediate disconnect: dedicated shutdown path.
```

## Resolved decisions

### `disconnect_with_timeout()` timeout start

The timeout starts when the event loop observes the disconnect barrier and begins the drain phase. It does not include
time spent waiting in the application request channel before the event loop observes the request.

The timeout bounds the graceful drain phase after the barrier is observed. If the drain completes before the deadline,
the event loop then attempts to write and flush MQTT `DISCONNECT`. If the drain deadline expires first, polling returns
`ConnectionError::DisconnectTimeout`, tears down the connection path, and MQTT `DISCONNECT` is not sent.

### `disconnect_now()` admission path

`disconnect_now()` should use a separate shutdown/control path rather than ordinary request-channel admission.

A method named `disconnect_now()` should not be blocked behind a full application request channel or a
flow-control-stalled publish queue. It should be modeled as a shutdown signal, not as another FIFO application
operation.

The separate path must stay narrow: only shutdown/abort control should use it. It must not become a general priority
lane for arbitrary requests.

Suggested semantics:

- `disconnect_now()` requests immediate connection termination.
- It does not wait for unresolved QoS 1/2 publish handshakes.
- It may bypass queued application work.
- If the implementation chooses to send MQTT `DISCONNECT`, that packet is terminal: after it is sent, no more MQTT
  control packets may be sent on that network connection.
- If the connection cannot send/flush `DISCONNECT` promptly, the transport may simply be closed/aborted.

### `SUBSCRIBE` / `UNSUBSCRIBE` and blocked QoS 1/2 publishes

`SUBSCRIBE` and `UNSUBSCRIBE` should be allowed to pass QoS 1/2 publishes that are blocked only by the publish
inflight / Receive Maximum window.

MQTT 5 Receive Maximum gates unacknowledged QoS 1/2 `PUBLISH` packets. It must not be used as a reason to delay
non-`PUBLISH` packets. `SUBSCRIBE` and `UNSUBSCRIBE` are non-`PUBLISH` control packets, so a full publish window should
not block them.

Packet identifier availability still applies. `SUBSCRIBE` / `UNSUBSCRIBE` may be blocked by packet-id exhaustion or
collision rules, but not merely by QoS 1/2 publish Receive Maximum.

Document the API consequence:

```text
rumqttc does not guarantee global wire FIFO across all API calls under publish flow-control pressure.
It preserves protocol-correct ordering and class-specific ordering:

- QoS 1/2 publishes preserve publish ordering subject to inflight limits.
- MQTT acknowledgement ordering is preserved.
- Tracked subscribe/unsubscribe requests preserve their own completion semantics.
- Non-PUBLISH control packets such as SUBSCRIBE, UNSUBSCRIBE, PINGREQ, AUTH, and DISCONNECT may be sent before
  earlier queued QoS 1/2 publishes that cannot currently be sent because the publish inflight window is full.
```

### QoS 0 publishes and blocked QoS 1/2 publishes

QoS 0 publishes should not pass blocked QoS 1/2 publishes by default.

MQTT 5 permits QoS 0 to be sent while QoS 1/2 Receive Maximum is full, because QoS 0 is not counted by Receive
Maximum. However, QoS 0 is still a `PUBLISH` packet, and MQTT does not require it to bypass earlier blocked publishes.

The less surprising rumqttc API contract is:

- Preserve FIFO among application publishes.
- Do not let later QoS 0 publishes overtake earlier QoS 1/2 publishes by default.
- Only non-`PUBLISH` protocol/control traffic bypasses publish inflight flow control.

### v4/v5 scheduler sharing

v4 and v5 should share the scheduler structure as much as practical.

MQTT 5 has Receive Maximum. MQTT 3.1.1 does not. But rumqttc already has local inflight limits for QoS 1/2 publish
flow control. From the scheduler's point of view, both versions need the same shape:

- QoS 1/2 publishes are flow-controlled.
- QoS 0 publishes are not part of that inflight accounting.
- Protocol-progress packets must continue to move.
- Control packets should not be accidentally blocked behind publish flow control.
- `DISCONNECT` must be terminal once sent.

Parameterize the source of the publish quota:

```text
v5: effective quota = min(local inflight limit, peer Receive Maximum)
v4: effective quota = local inflight limit
```

The scheduler policy should be shared for consistency across users, tests, and maintainers.

## Suggested implementation steps

1. Add scheduler tests before refactoring:
   - Full receive maximum must block further QoS 1/2 `PUBLISH`.
   - Full receive maximum must not block `SUBSCRIBE`, `UNSUBSCRIBE`, `PINGREQ`, `AUTH`, `DISCONNECT`, `PUBACK`,
     `PUBREC`, `PUBREL`, or `PUBCOMP` once those packets are observable to the event loop.
   - `PUBACK`, `PUBREC`, and `PUBREL` ordering rules are preserved.
   - `DISCONNECT` remains terminal; no packet is sent after it.
   - Graceful disconnect drains the intended barrier set before sending `DISCONNECT`.
   - Immediate disconnect does not wait for unresolved publish state.
2. Centralize request classification:
   - Identify whether a request is QoS 1/2 publish-flow-controlled, QoS 0 publish, protocol-progress, or control.
   - Ensure manual ack requests are not accidentally held behind publish inflight limits.
3. Split queueing from sending:
   - Request channel admission may remain a bounded API backpressure mechanism.
   - Event-loop scheduling should not equate a full publish inflight window with "do not inspect or send anything".
4. Preserve retransmission and reconnect behavior:
   - `clean()` must retain the right pending order for unacknowledged publishes and QoS 2 continuations.
   - Do not replay inbound ack requests from a previous connection in a way that confuses the broker.
   - MQTT 5 topic alias replay repair must remain correct for pending publishes.
5. Update docs:
   - Document `try_*` as request-channel admission APIs, not MQTT wire-flow guarantees.
   - Document graceful vs immediate disconnect semantics and timeout start semantics.
   - Note that transport abort/drop is not MQTT `DISCONNECT`.
